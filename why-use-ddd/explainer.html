THE PROBLEM
软件复杂性的代价
每个开发团队都在经历这样的痛苦：系统越来越复杂，没人敢改代码，改一处崩三处。
❌ 信息偏差

产品经理说"下单"，程序员想到的是 INSERT 一条记录。同一件事，两种理解。每一次信息转换的偏差，就是一个潜在的 Bug。

❌ 高度耦合

系统各部分高度耦合，改一处，处处崩。一环挂全挂，谁都不敢动这坨代码。

❌ 无法演进

三十万行的单体应用，改一个需求，整个系统都在颤抖。没有边界，就没有安全感。

根因：没有统一语言，没有边界。软件复杂性的本质不是技术复杂度，而是业务复杂度。DDD 就是为此而生的。
BENEFIT 1
好处一：统一语言
DDD 引入通用语言（Ubiquitous Language），消除产品、开发、测试之间的理解偏差。
❌ 没有通用语言
// 产品经理说："下单" // 程序员写了： function saveOrderToDB(data) { db.insert('orders', data); // 库存在哪？通知在哪？ // 没人知道... }
✅ DDD 通用语言
// 通用语言：placeOrder class OrderService { async placeOrder(cmd) { order.confirm(); // 业务规则 await payment.charge(); domainEvent.publish(); } }
核心价值：代码里的命名直接就是业务概念。placeOrder 而不是 saveToDatabase。改下单规则？改 Order 聚合根就行，不用在二十个文件里找逻辑。
BENEFIT 2
好处二：划分边界，分而治之
DDD 用限界上下文（Bounded Context）把系统切成独立的块，每个块有自己的领域模型和业务规则。
❌ 30 万行单体应用

订单逻辑、库存代码、支付逻辑、通知代码全部混在一起。改一个需求，整个系统都在颤抖。

订单 库存 支付 通知 促销 用户

全部耦合在一起，无法独立演进

✅ 限界上下文划分
订单上下文
下单 · 改价 · 取消
库存上下文
扣减 · 锁定 · 释放
支付上下文
扣款 · 退款 · 对账
通知上下文
短信 · 邮件 · 推送

独立演进 · 互不影响 · 明确接口通信

STRATEGIC DESIGN
战略设计：三个域
不是所有代码都同等重要。DDD 将业务分为核心域、支撑域和通用域，指导资源分配。
★ 核心域

公司的护城河

  • 电商的订单和支付
  • 物流的路线规划
  • 推荐的算法引擎
→ 配最强工程师，精细建模
⚙ 支撑域

必要但非差异化

  • 报表系统
  • 基础数据管理
  • 内部工具
→ 适度投入，够用就行
□ 通用域

直接用成熟方案

  • 用户认证 / OAuth
  • 权限管理 / RBAC
  • 日志 / 监控
→ 不要重复造轮子
关键洞察：战略层面最重要的决策是识别核心域。把最优秀的工程师放在核心域，因为这是公司的核心竞争力所在。
STRATEGIC DESIGN
限界上下文 & 上下文映射
限界上下文定义领域的边界。同一个上下文内术语含义一致，跨上下文则可能完全不同。

同一个词，不同含义

销售上下文中的"商品"
价格 促销 描述 分类

面向客户，关注销售属性

仓储上下文中的"商品"
SKU 数量 仓位

面向仓库，关注库存属性

上下文映射模式

防腐层 (ACL)

隔离外部系统，翻译外部模型为自己的模型，防止外部概念污染领域模型。

开放主机服务 (OHS)

提供标准协议（REST / gRPC），多个消费者统一接入，降低集成成本。

发布语言 (PL)

用统一的数据格式（JSON Schema / Protobuf）通信，各上下文共享语义。

TACTICAL DESIGN
战术设计：五个核心构建块
实体、值对象、聚合、领域服务、仓储 — 这五个构建块组合使用，构成完整的领域模型。
实体 Entity

有唯一 ID 的对象，通过 ID 判等。

例如：订单、用户、商品

// 通过 ID 判等 order1.equals(order2) → by id
值对象 Value Object

没有唯一 ID，通过属性值判等。不可变。

例如：地址、金额、日期

// 通过值判等 money1.equals(money2) → by value
聚合 Aggregate

一组相关对象的集合，作为数据修改的单元。聚合根保护业务规则不变。

Order → OrderItem[]
领域服务

不属于任何实体的业务逻辑。跨聚合的操作放这里。

transfer(from, to, amt)
仓储 Repository

为聚合提供持久化抽象。领域层不用关心底层存储。

save() findById()

六边形架构（端口 + 适配器）

六边形架构的核心思想：领域层在中心，通过端口（接口）与外部通信，适配器负责具体的技术实现。领域层不依赖任何基础设施，可以独立测试和演进。

代码示例：端口与适配器

领域层定义端口（接口），基础设施层提供适配器（实现）。这就是依赖反转原则（DIP）。

📐 领域层：定义端口（接口）
// 输出端口 — 仓储接口 interface OrderRepository { Order findById(OrderId id); void save(Order order); } // 输出端口 — 事件发布 interface EventPublisher { void publish(DomainEvent event); } // 输入端口 — 应用服务 class PlaceOrderService { constructor( private orderRepo: OrderRepository, private events: EventPublisher ) {} execute(cmd: PlaceOrderCmd) { const order = Order.create(cmd); this.orderRepo.save(order); this.events.publish( new OrderPlaced(order.id) ); } }
🔧 基础设施层：实现适配器
// 适配器 — MySQL 实现仓储 class MySqlOrderRepository implements OrderRepository { findById(id: OrderId): Order { const row = await db.query( 'SELECT * FROM orders WHERE id=?', [id.value] ); return OrderMapper.toDomain(row); } save(order: Order): void { const row = OrderMapper.toRow(order); await db.insert('orders', row); } } // 适配器 — Kafka 事件发布 class KafkaEventPublisher implements EventPublisher { publish(event: DomainEvent) { await kafka.send( event.topic, event.payload ); } }

依赖方向

关键原则：依赖指向内部。外层依赖内层，内层绝不依赖外层。

层职责典型技术依赖方向
驱动端适配器 接收外部请求，调用应用服务 Spring MVC, Express, GraphQL → 依赖应用层
应用层 编排用例，不包含业务规则 Application Service → 依赖领域层
领域层（核心） 实体、值对象、聚合、领域服务 纯 Java/Kotlin/TS ★ 零依赖
被驱动端适配器 实现领域层定义的端口接口 JPA, MyBatis, Redis, Kafka → 实现领域层接口
🔑 核心价值：领域逻辑不依赖任何框架和基础设施。你可以把 MySQL 换成 MongoDB，把 Kafka 换成 RabbitMQ，领域层代码一行都不用改。这就是六边形架构的威力 — 可替换、可测试、可演进。

总结与实战建议
DDD 的核心价值：统一语言、划分边界、业务驱动。
统一语言

消除歧义，产品、开发、测试用同一种语言沟通

划分边界

限界上下文隔离复杂度，大系统变小系统

业务驱动

实战建议

⚠️ DDD 不是银弹

简单 CRUD 不需要 DDD，别过度设计

业务复杂度够高，DDD 才是你的核武器

👥 业务专家参与

DDD 不是开发团队自己玩的事，业务专家必须参与

🏆 掌握 DDD，是从码农到架构师的必经之路。让你的软件，真正反映业务的本质。
