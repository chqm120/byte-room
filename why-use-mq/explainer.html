PROBLEM

同步耦合的痛点

一个下单请求，同步调用库存、支付、通知、积分四个服务。每个服务 50ms，总耗时 200ms。任何一个服务慢或挂掉，整个下单请求就失败。这就是典型的同步耦合问题。

WHAT IS MQ

Message Queue — 消息队列

MQ 是一个独立部署的中间件，本质上是一个以消息为载体的通信协议。生产者把消息丢进队列，消费者从队列取消息处理。生产者与消费者完全解耦，互不感知。

CORE VALUE · 1

异步解耦

下单服务只需把消息丢进 MQ，1ms 就返回。库存、支付、通知服务各自订阅消息，独立处理。新增风控服务只需要订阅消息，下单服务代码一行不改。

❌ 同步 · 紧耦合
// 下单服务代码 createOrder(req) { order = db.save(req); stockSvc.deduct(order); // 50ms paySvc.create(order); // 50ms notifySvc.send(order); // 50ms pointSvc.add(order); // 50ms return order; // 总共 200ms }
✅ 异步 · 松耦合
// 下单服务代码 createOrder(req) { order = db.save(req); mq.send("ORDER_CREATED", order); // 1ms 返回！ return order; } // 库存服务（独立订阅） @Subscribe("ORDER_CREATED") onOrder(event) { deduct(event); }
💡 松耦合的威力：下单服务不知道有哪些消费者，将来新增任何下游服务，都不需要修改下单服务的代码。这就是开闭原则 — 对扩展开放，对修改关闭。
响应时间对比 200ms → 1ms

流量削峰

秒杀场景：十万请求在同一秒涌入。没有 MQ，数据库直接崩溃。有了 MQ，消费者按自己的能力匀速消费，系统稳如泰山。

🔴 无 MQ · 流量直打数据库

请求量

匀速 1000/s
稳定消费
⚡ 削峰本质：把尖锐的流量脉冲变成匀速的细水长流。队列容量就是缓冲池，消费者的处理能力就是阀门。只要峰值总量不超过队列容量，系统就不会崩。

消息存储原理

RocketMQ 高性能的秘密：所有 Topic 的消息共享一个 CommitLog 顺序写文件，配合 ConsumeQueue 索引实现极速读取。

KILLER FEATURE

事务消息

RocketMQ 的王牌特性。解决分布式事务难题：本地数据库操作和发送消息，要么都成功，要么都失败。

事务消息执行流程
1
发送 Half 消息
Producer 发送一条对 Consumer 不可见的半消息到 Broker
2
执行本地事务
Broker 返回 Half 消息成功后，Producer 执行本地数据库事务
3
Commit 或 Rollback
本地事务成功 → Commit（消息对 Consumer 可见）
本地事务失败 → Rollback（消息丢弃）
4
事务回查（兜底）
如果 Broker 长时间没收到 Commit/Rollback，主动回查 Producer 的本地事务状态
// RocketMQ 事务消息代码示例 TransactionListener listener = new TransactionListener() { // 执行本地事务 public LocalTransactionState executeLocalTransaction(Message msg, Object arg) { try { orderService.createOrder(arg); // 本地数据库操作 return LocalTransactionState.COMMIT_MESSAGE; } catch (Exception e) { return LocalTransactionState.ROLLBACK_MESSAGE; } } // 事务回查 public LocalTransactionState checkLocalTransaction(MessageExt msg) { return orderService.isOrderCreated(msg) ? LocalTransactionState.COMMIT_MESSAGE : LocalTransactionState.ROLLBACK_MESSAGE; } };
💡 两阶段提交的 MQ 实现：Half 消息就是 prepare 阶段，Commit/Rollback 就是 commit/rollback 阶段，事务回查就是超时补偿。和数据库 XA 事务原理一样，但性能高出几个数量级。
CORE VALUE · 3

消息可靠传递

在分布式系统中，网络不可靠，服务会宕机。MQ 通过持久化、确认机制、重试策略，确保消息从生产到消费不丢失。

1
Producer
发送消息
等待 ACK 确认
失败自动重试
2
Broker
写入 CommitLog
同步刷盘落磁盘
主从复制到 Slave
3
Consumer
拉取消息处理
成功后发 ACK
失败则重新投递
📤
生产端保障
  • 同步发送 + 重试
  • 发送失败入死信队列
  • 事务消息保障
💾
Broker 保障
  • 同步刷盘（SYNC_FLUSH）
  • 主从复制（SYNC_MASTER）
  • 多副本 Dledger 模式
📥
消费端保障
  • 处理成功后才 ACK
  • 消费失败自动重试
  • 幂等性消费防重复
💡 对于支付、交易等核心业务，必须开启同步刷盘 + 同步复制。牺牲一点性能，换来消息绝不丢失。
🔄 高可用机制：主从复制与 Dledger
传统主从模式
  • Master 负责读写
  • Slave 只读，异步/同步复制
  • Master 挂了需人工切换
Dledger 模式（推荐）
  • 基于 Raft 协议
  • Leader 挂了自动选主，秒级切换
  • 数据强一致，生产推荐方案
PRACTICE

实战场景

MQ 在互联网系统中的应用无处不在。以下是最经典的三个实战场景。

场景一
订单系统

用户下单后，订单服务发送「订单创建」消息。库存、支付、物流、通知服务各自订阅处理，互不干扰。

下单服务 → MQ 库存服务 ← MQ 支付服务 ← MQ 物流服务 ← MQ 通知服务 ← MQ
场景二
秒杀系统

海量请求涌入时，先写入 MQ 缓冲，后端按最大能力匀速消费。超库存请求直接拒绝，用户看到「排队中」。

10w 请求涌入 MQ 缓冲队列 匀速 1000/s 消费 数据库稳如泰山
场景三
数据同步

通过监听 MySQL Binlog 把数据变更发到 MQ，下游 Elasticsearch、Redis、数据仓库各自订阅，完成数据同步。

MySQL Binlog Canal → MQ ES 索引更新 Redis 缓存更新 数据仓库同步
💡 共性模式：所有场景的本质都是同一个模式 — 上游产生事件，MQ 传递事件，下游处理事件。系统之间通过消息实现最终一致性，而不是强一致性。
