PROBLEM

高并发的痛点

当十万并发请求直接打到数据库，MySQL 需要查磁盘、加锁、写日志，每一步都是毫秒级的开销。最终结果是响应超时、连接耗尽、系统崩溃。

磁盘 IO
~10ms
每次查询至少一次磁盘随机读
锁等待
~50ms
行锁、表锁竞争加剧延迟
连接池
耗尽
每个连接占内存，通常上限几千
💡 关键洞察：磁盘 IO 比内存操作慢 10 万倍。这就是为什么我们需要一个纯内存的中间层。
WHAT IS REDIS

Redis — Remote Dictionary Server

Redis 是一个开源的内存键值数据库。所有数据存储在内存中，读写操作都是纳秒级。它不是要替代 MySQL，而是作为高速缓存层，拦截绝大多数读请求。

💿 磁盘 · MySQL
50ms
典型查询耗时
🧠 内存 · Redis
0.5ms
典型查询耗时
性能提升 100×
// 典型的缓存读取流程
String value = redis.get("user:1001");
if (value == null) {
  value = mysql.query("SELECT * FROM user WHERE id=1001");
  redis.setex("user:1001", 3600, value); // 缓存1小时
}
SINGLE THREADED

单线程为什么更快？

Redis 核心命令处理是单线程的，但这并不意味着它不能并发处理请求。通过 IO 多路复用（epoll），单个线程就能高效处理数万并发连接。

🏃‍♂️
Redis · 单线程
  • ✅ 无需锁竞争
  • ✅ 无上下文切换开销
  • ✅ 无多线程同步复杂性
  • ✅ 内存操作 ~100ns，足够快
  • ✅ epoll 处理数万并发连接
👥
传统 DB · 多线程
  • ❌ 锁竞争消耗 CPU
  • ❌ 线程上下文切换 ~1000ns
  • ❌ 同步机制复杂
  • ❌ 磁盘 IO 阻塞线程
  • ❌ 线程数越多效率可能越低
💡 核心逻辑：内存操作 ~100ns，线程切换 ~1000ns。当操作本身足够快时，多线程的协调开销反而成了瓶颈。
// Redis 事件循环伪代码
while (!stopped) {
  events = epoll_wait(fd, timeout); // 监听多个连接
  for (event : events) {
    processCommand(event.client); // 执行命令（单线程）
    replyToClient(event.client);
  }
}
DATA STRUCTURES

六种基础数据结构

Redis 不只是简单的 key-value 缓存。它提供了丰富的数据结构，每种都有独特的使用场景。

String
字符串
最基础的类型，值可以是字符串、数字或二进制数据。原子递增/递减操作使其天然适合计数器。
SET session:abc123 "user:1001" EX 1800  // 会话管理
INCR article:9527:views                  // 文章浏览量
SETNX lock:order:5001 "worker-1" EX 30    // 分布式锁
Hash
哈希
存储字段-值映射，特别适合对象数据。可以单独读写某个字段，无需序列化整个对象。
HSET user:1001 name "Alice" age 28 city "Beijing" email "alice@example.com"
HGET user:1001 name    // → "Alice"
HINCRBY user:1001 login_count 1 // 登录次数 +1
List
列表
有序的字符串链表，支持两端推入/弹出。天然的消息队列实现，也可用于时间线和最新列表。
LPUSH queue:orders "order:5001"  // 生产者推入
RPOP queue:orders            // 消费者弹出（FIFO）
LRANGE timeline:user:1001 0 9 // 获取最新动态前10条
Set
集合
无序、不重复。支持交集、并集、差集运算。
标签系统 共同好友 去重
SINTER user:1001:friends user:1002:friends
// → 共同好友
ZSet
有序集合
每个元素关联一个 score，自动排序。底层用跳表实现。
排行榜 延迟队列 滑动窗口
ZADD leaderboard 9850 "Alice"
ZREVRANGE leaderboard 0 2 WITHSCORES
// → 前三名
更多高级结构
Bitmap — 签到、在线状态 HyperLogLog — UV 统计（误差 0.81%） GEO — 附近的人 Stream — 消息队列（支持消费者组）
INTERNALS

自适应编码优化

Redis 会根据数据量和数据类型自动选择最优的底层编码，在小数据量时极度节省内存，在大数据量时保持高性能。

数据结构 小数据编码 大数据编码 切换条件
Hash ziplist / listpack hashtable > 512 字段或值 > 64B
ZSet ziplist / listpack skiplist + hashtable > 128 元素或值 > 64B
Set intset hashtable 非纯整数或 > 512 元素
List ziplist / listpack linkedlist / quicklist 元素 > 64B 或 > 1024 个
💡 实际效果：一个只有 5 个字段的用户 Hash，用 ziplist 只占 ~200 字节，用 hashtable 则需要 ~1KB+。百万级小对象场景下节省数百 MB 内存。
PERSISTENCE

持久化机制

内存数据断电即失。Redis 提供两种持久化机制，生产环境通常混合使用。

📸
RDB 快照
在指定时间间隔内，将内存数据集的快照写入磁盘（.rdb 文件）。使用 fork + COW 机制，不阻塞主线程。
恢复速度快 文件紧凑 适合备份
⚠ 两次快照之间的数据可能丢失
# redis.conf
save 900 1    # 900s内至少1次修改
save 300 10  # 300s内至少10次修改
save 60 10000 # 60s内至少10000次修改
📝
AOF 追加日志
记录每一条写命令到日志文件（.aof）。重启时重放日志即可恢复数据。支持 always / everysec / no 三种刷盘策略。
数据更安全 最多丢1秒 可读性强
⚠ 文件较大，恢复较慢；需要定期 rewrite 压缩
# redis.conf
appendonly yes
appendfsync everysec # 每秒刷盘（推荐）
✅ 生产推荐：RDB + AOF 混合持久化。AOF 用于数据安全，RDB 用于快速恢复和定期备份。Redis 4.0+ 支持 AOF rewrite 时生成混合格式。
CACHE PROBLEMS

三大缓存问题

引入缓存后，需要面对三个经典问题。核心原则是：不能让请求直接穿透到数据库。

穿透
缓存穿透
查询一个根本不存在的数据（如 id=-1 的用户），缓存永远不命中，每次请求都打到数据库。恶意攻击者可以利用这一点，用不存在的 key 压垮数据库。
🛡 解决方案：
缓存空值 — 查不到也缓存 null，设短 TTL 布隆过滤器 — 在缓存前面拦截不存在的 key 参数校验 — 入口拦截非法参数
击穿
缓存击穿
某个超级热点 key（如首页轮播数据）在高并发下突然过期，同一瞬间数千个请求全部打到数据库，导致数据库瞬时负载飙升。
🛡 解决方案：
互斥锁 — 只让一个线程重建缓存，其他等待 永不过期 — 逻辑过期 + 异步更新 热点预加载 — 大促前提前预热
雪崩
缓存雪崩
大量 key 在同一时间集体过期，或者 Redis 实例宕机，导致原本被缓存拦截的请求全部涌向数据库。
🛡 解决方案：
随机 TTL — 过期时间 + random(0, 300) 多级缓存 — L1 本地 + L2 Redis + L3 DB 熔断降级 — 数据库压力过大时返回默认值 Redis 高可用 — 哨兵 / Cluster
