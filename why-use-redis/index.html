SINGLE THREADED
单线程为什么更快？
🛒 超市比喻
🏃‍♂️
Redis · 单收银员
手速极快（内存操作）
商品都在手边（无 IO 等待）
不用和别人协调（无锁）
一个接一个处理（无上下文切换）
✓ 高效简洁
👥
传统 DB · 多收银员
需要频繁沟通协调（锁竞争）
排队等仓库出货（磁盘 IO）
切换窗口有开销（上下文切换）
多线程同步复杂度高
✗ 协调开销大
💡 技术本质
内存操作 ~100ns，线程切换开销 ~1000ns
线程切换比操作本身还慢 → 多线程反而更慢
IO 多路复用 · epoll
Redis 使用 epoll 监听多个客户端连接。单线程通过事件循环处理所有 IO，非阻塞高效。

CACHE PROBLEMS
三大缓存问题
穿透
缓存穿透
查询不存在的数据 → 缓存永远不命中 → 全部打到数据库
✓ 缓存空值 ✓ 布隆过滤器
击穿
缓存击穿
热点 key 突然过期 → 大量请求瞬间涌入数据库
✓ 互斥锁 ✓ 永不过期
雪崩
缓存雪崩
大量 key 同时过期 → 数据库瞬间被压垮
✓ 过期时间加随机值 ✓ 多级缓存
💡 核心思路：不要让所有请求同时穿透到数据库
